# Описание решения — Кредитный скоринг (прогноз дефолта)

## 1. Постановка задачи

Задача — предсказать вероятность выхода клиента в дефолт по кредиту (неплатёж более 3 месяцев в течение года) по истории его кредитных продуктов. Данные предоставлены в long-формате: на каждого клиента (`id`) приходится от 1 до 55 строк — по одной на каждый кредитный продукт в его истории. Метка дефолта (`flag`) — одна на клиента. Метрика — ROC-AUC. Дисбаланс классов: ~3.55% дефолтов. Train/test разделены по времени (более поздние `id` — тест).

## 2. Архитектура решения

Решение построено на **двух независимых представлениях** одних и тех же данных, объединённых rank-блендом:

```
                          ┌─────────────────────────────────┐
                          │    Исходные данные (long)       │
                          │  18.3M строк / 2.1M клиентов    │
                          │  59 признаков × rn продуктов    │
                          └──────────┬───────────┬──────────┘
                                     │           │
                    ┌────────────────┘           └────────────────┐
                    ▼                                             ▼
          Представление 1:                              Представление 2:
          АГРЕГАТЫ НА ID                              ПОСЛЕДОВАТЕЛЬНОСТИ
          ─────────────────                           ─────────────────
          groupby(id) →                               Сырые строки →
          182 признака на клиента                     (rn₁, rn₂, ..., rnₖ)
          (mean/max/min/std/counts)                   59 кодов на шаг
                    │                                             │
            ┌───────┴───────┐                                     │
            ▼               ▼                                     ▼
        LightGBM        CatBoost                         Embedding bi-GRU
        OOF 0.7548      OOF 0.7548                       OOF 0.7798
            │               │                                     │
            └───────┬───────┘                                     │
                    │                                             │
                    └──────────────┬──────────────────────────────┘
                                   ▼
                          RANK-BLEND (OOF-оптимизация)
                          Emb-GRU 84% + CatBoost 10% + LGBM 6%
                          ─────────────────────────────────────
                          OOF ROC-AUC = 0.7808
```

### 2.1. Ветка агрегатов → GBDT

**Идея:** свернуть историю каждого клиента в один вектор фиксированной длины (mean/max/min/std/counts по группам признаков), после чего применить градиентный бустинг.

- **Агрегация** строго внутри каждого `id` (нет cross-id утечки). 8 блоков: volume, ordinal_stats, delinquency, flags, payment_status, categorical_diversity, categorical_counts, last_product → **182 признака**.
- **LightGBM** (`scale_pos_weight ≈ 27` для компенсации дисбаланса, 5 фолдов, early stopping по fold-validation). OOF ROC-AUC = **0.7548**.
- **CatBoost** (те же фичи/фолды, `auto_class_weights=Balanced`). OOF ROC-AUC = **0.7548**.

### 2.2. Ветка последовательностей → Embedding bi-GRU

**Идея:** не сворачивать историю в агрегаты, а подать последовательность продуктов в рекуррентную сеть, которая сама выучит полезные паттерны.

- **Все 59 признаков — категориальные коды** (бины 0–19, статусы 0–4, флаги 0/1), поэтому каждый получает собственный обучаемый **эмбеддинг** (реализовано через общую таблицу `nn.Embedding` + per-column offsets, размер словаря = `max(train, test) + 1`).
- Эмбеддинги конкатенируются по каждому таймстепу → вход размерности 59 × `emb_dim` = 472.
- **Двунаправленный GRU** (hidden=128) читает последовательность; финальные скрытые состояния обоих направлений конкатенируются → MLP (128 → 1) → логит дефолта.
- Паддинг замаскирован через `pack_padded_sequence`; последовательности усечены до 40 последних продуктов (наиболее свежие).
- Обучение на MPS (Apple M3 Pro), `BCEWithLogitsLoss` с `pos_weight`, early stopping по fold-validation (patience=2).
- OOF ROC-AUC = **0.7798** — главный рычаг качества (+0.025 к GBDT).

### 2.3. Ансамбль

**Метод:** rank-percentile blend. Предсказания каждой модели ранг-нормализуются (0..1), чтобы выравнять масштабы (GBDT-вероятности vs NN-сигмоида). Веса подбираются Nelder-Mead оптимизацией по OOF ROC-AUC (неотрицательные, сумма = 1). OOF/test выравниваются по **значению `id`**, а не по позиции в массиве.

Финальные веса: **Embedding bi-GRU 84% + CatBoost 10% + LightGBM 6%**. OOF ROC-AUC = **0.7808**.

Вес эмбеддинговой модели доминирует — GBDT добавляет лишь +0.001, но стабилизирует.

## 3. Ключевые решения и обоснования

### Почему два представления, а не одно?

Агрегаты и последовательности извлекают **разный сигнал**: агрегаты хорошо ловят «средний профиль» клиента, последовательности — паттерны развития (ухудшение/улучшение платёжной дисциплины). Их ранговая корреляция достаточно низка для прироста в ансамбле.

### Почему эмбеддинги, а не сырые числа?

Все 59 признаков — целочисленные коды (бины, статусы, флаги). Если подавать их как числа, модель предполагает линейный порядок (bin 5 > bin 3), что верно для ordinal-бинов, но не для категориальных кодов (`enc_loans_credit_type=2` не «больше» 1). Эмбеддинги позволяют каждому коду иметь собственное обучаемое представление, и модель сама выучивает семантику. Эффект: +0.025 ROC-AUC к scaled-numeric GRU на полном объёме.

### Почему rank-blend, а не raw-probability?

Масштабы предсказаний GBDT и NN различаются (GBDT с `scale_pos_weight` даёт завышенные вероятности, NN — сигмоиду). Rank-нормализация приводит всех к единой шкале перцентилей, что делает блендинг инвариантным к масштабу. ROC-AUC — ранговая метрика, поэтому это корректно.

### Почему GBDT-агрегаты вышли на плато?

Ablation-эксперименты (`v2`, `v3`) показали, что производные агрегаты (trend, delinq_share, term_gaps, sentinel-aware сроки) не добавляют сигнала поверх baseline-набора из 182 признаков. Причина: GBDT нативно строит любые пороги и взаимодействия по ordinal-бинам — ручные комбинации не дают ему ничего нового. Весь дополнительный сигнал — в **порядке** продуктов (последовательностях), который агрегаты уничтожают.

### Почему StratifiedKFold, а не TimeSeriesSplit?

После агрегации на каждый `id` ровно одна строка с одной меткой. `StratifiedKFold(5, seed=42)` по `id` корректен: утечка строк невозможна, стратификация важна при дисбалансе 3.55%. Однако платформенный тест — временной (поздние `id`), поэтому дополнительно считается time-holdout (поздние 20% `id`) для оценки разрыва.

| дата | сабмит | OOF | Public AUC | разрыв OOF→Public |
|---|---|---|---|---|
| 06-18 | LightGBM baseline | 0.7548 | 0.75591 | +0.0011 |

### Временной дрейф — опасения не подтвердились

| метрика | значение |
|---|---|
| random-CV OOF (baseline) | 0.7548 |
| time-holdout (поздние 20% id) | 0.7295 |
| **реальный Public (baseline)** | **0.75591** |

Time-holdout прогнозировал просадку −0.025 на тесте, но **реальный Public совпал с random-CV OOF (даже чуть выше)**. Значит «поздние 20% id» как holdout — *не* репрезентативны для тестового распределения (это просто более трудный сегмент train), а платформенный тест по сложности близок к среднему по train.

**Практический вывод:** random-CV OOF — надёжный, слегка консервативный прокси лидерборда (разрыв +0.001…+0.002 в плюс). Отбор моделей и приёмка приростов ведутся по OOF; time-holdout оставлен как диагностика, но не как критерий отбора.

## 5. Известные ограничения

1. **Доверие к OOF (разрешено):** изначальное опасение по временному дрейфу (time-holdout −0.025) **не подтвердилось** — реальный Public совпал с random-CV OOF (даже +0.001…+0.002). OOF можно доверять как критерию отбора. Остаточный риск: распределение приватного теста может отличаться от публичного, поэтому не переобучаемся под Public.

2. **Одиночный seed у Embedding bi-GRU:** нейросеть обучена с одним seed=42. Multi-seed усреднение (3–5 seed) обычно даёт +0.001–0.002 и стабилизирует.

3. **Нет stacking через фичи:** OOF-предсказания sequence-модели не используются как входные фичи для GBDT (stacking). Это потенциальный рычаг, но требует отдельного CV-слоя.

4. **Калибровка не проводилась:** ROC-AUC — ранговая метрика, поэтому калибровка не влияет на скор. Но для бизнес-применения (пороги принятия решений) предсказания не калиброваны.

5. **Sentinel-кодирование:** `pclose_flag=1` → `pre_pterm=4` (sentinel внутри ordinal-шкалы). GBDT обрабатывает это нативно (флаги есть в фичах), но для линейных моделей или интерпретации это может быть проблемой.

## 6. Перспективные пути улучшения

Поскольку OOF — надёжный прокси Public (разрыв +0.001…+0.002), приоритет — у направлений, повышающих OOF. Доминирующая модель — Embedding bi-GRU (вес 84%, 0.7798), GBDT на плато (0.755). Значит основной headroom — в sequence-ветке.

### Сводная таблица (по приоритету = эффект × вероятность)

| # | направление | ожидаемый Δ OOF | вероятность | усилия | компонента |
|---|---|---|---|---|---|
| 1 | Архитектура/ёмкость sequence-модели | +0.003…+0.012 | высокая | средние | seq |
| 2 | Transformer-энкодер над продуктами | +0.005…+0.015 / диверсификация | средняя | высокие | seq |
| 3 | Multi-seed averaging bi-GRU | +0.001…+0.003 | очень высокая | низкие (×компьют) | seq |
| 4 | Stacking: OOF-эмбеддинги → GBDT | +0.003…+0.010 | средняя | средние | оба |
| 5 | Диверсификация ансамбля (XGB + seed-модели) | +0.002…+0.005 | высокая | низкие | ens |
| 6 | Обогащение входов sequence-модели | неизвестно | средняя | средние | seq |
| 7 | HPO GBDT (Optuna) | +0.000…+0.001 | низкая | низкие | gbdt |

### Tier 1 — усиление доминирующей sequence-модели

**1. Архитектура и ёмкость bi-GRU (главный рычаг).**
Текущая модель скромная: 1 слой, hidden=128, emb_dim=8, max_len=40, пуллинг по последнему состоянию. Эксперименты (каждый — изменение конфига + полный прогон ~2ч, отбор по OOF):
- `max_len` 40 → 55: сейчас усекаем до 40 последних продуктов, у части клиентов до 55 — теряем хвост истории. Дёшево проверить.
- `hidden` 128 → 256, `layers` 1 → 2: больше ёмкости.
- `emb_dim` 8 → 16: богаче представление кодов.
- **Пуллинг:** заменить last-hidden на attention-pooling или mean+max по всем таймстепам (часто заметно лучше для переменной длины).
- LR-расписание (cosine/warmup) + больше эпох.
Ожидание: для доминирующей модели даже относительный +0.005 даёт ощутимый Public. Риск: переобучение при росте ёмкости — контролировать по OOF.

**2. Transformer-энкодер над последовательностью продуктов.**
2–4 слоя multi-head attention поверх тех же эмбеддингов + позиционное кодирование, пуллинг через [CLS]/attention. Attention лучше ловит связи между непоследовательными продуктами, чем GRU. Эксперимент: реализовать `TransformerEncoder`-вариант в `models/sequence.py`, обучить, сравнить OOF. Даже при паритете с GRU — сильный диверсификатор для ансамбля (другая индуктивная предпосылка → низкая корреляция).

**3. Multi-seed averaging.**
Обучить bi-GRU на seed ∈ {42, 7, 13, 101, 2024}, усреднить ранги OOF/test. Снижает дисперсию NN (у одиночной модели fold-std ≈ 0.0024). Самый надёжный «дешёвый» прирост; ограничение — компьют (5 × ~2ч). Эксперимент: цикл по seed в конфиге + `07_ensemble` по 5 прогонам (или прямое усреднение в скрипте).

### Tier 2 — объединение представлений и диверсификация

**4. Stacking: эмбеддинги bi-GRU как фичи для GBDT.**
Извлечь пуллинг-представление bi-GRU (предпоследний слой, ~256-d) для каждого `id` — **leak-safe**: эмбеддинг клиента берётся из той fold-модели, что на нём не обучалась (OOF). Подать эти 256 признаков в LightGBM/CatBoost вместе со 182 агрегатами. Так сигнал последовательностей попадает в одну GBDT-модель, а не только через бленд. Эксперимент: доработать `sequence.py` для дампа OOF/test представлений → новый feature set → `03_train`. Ожидание: +0.003…+0.010; риск — переобучение GBDT на эмбеддингах (регуляризация, отбор).

**5. Диверсификация ансамбля.**
Текущий бленд — 3 модели, но GBDT почти дублируют друг друга (corr высокая). Добавить разнородные члены: XGBoost на агрегатах (в requirements, не задействован), GBDT-на-эмбеддингах (#4), Transformer (#2), seed-модели (#3) → 6–8 кандидатов → перейти от ручного rank-mean к **hill-climbing отбору** членов по OOF. Ожидание: +0.002…+0.005 поверх текущего ансамбля; диверсификация уже доказала эффект (v1→v2 дал +0.016 за счёт смены представления).

### Tier 3 — низкий приоритет

**6. Обогащение входов sequence-модели.** Дельты между соседними продуктами (динамика), обучаемое позиционное кодирование, отдельная числовая ветка для истинно-ordinal колонок. Эффект неизвестен.

**7. HPO GBDT (Optuna).** GBDT на плато и весит 6–10% в ансамбле — потолок ~0.755. Ожидание +0.000…+0.001, низкий приоритет.

**Что НЕ делать:** калибровка (ROC-AUC ранговая), удаление выбросов/нормализация для GBDT (нет эффекта), подгонка под Public (риск переобучения под публичную часть теста; доверяем OOF).

### Дисциплина экспериментов

- Все прогоны — на **тех же 5 фолдах** (seed=42), отбор по OOF (он надёжен), сравнение через `scripts/05_compare.py`.
- Приёмка прироста — только если Δ OOF > fold-std (~0.0025) или подтверждён ансамблем.
- Загрузки на платформу — экономно (≤3/день): грузим только при значимом росте OOF.
- Рекомендованный порядок: **#3 (multi-seed) параллельно с #1 (архитектура)** → **#4 (stacking)** → **#5 (диверсификация ансамбля)**.
